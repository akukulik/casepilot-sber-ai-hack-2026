# Демонстрационный сценарий CasePilot

## Выбранный кейс

`VAL-DC-002` — закрытие счёта заблокировано техническим отрицательным
остатком после начисления комиссии. Это самый наглядный и стабильный сценарий:
поиск находит сильный аналог `HIST-DC-001` (`score=0.8148`), план содержит три
шага и одну экспертизу, mock-корректировка устраняет блокер, а финальное решение
остаётся за сотрудником.

## Что вводить в Chat

Первое сообщение:

> VAL-DC-002

Агент должен показать без служебного JSON:

- ID и тему кейса;
- краткое понимание проблемы и ключевые признаки;
- основной сценарий закрытия счёта по техническому остатку;
- три шага: проверка состояния, `account_balance_analysis`, итоговая проверка
  возможности закрытия;
- одну необходимую экспертизу;
- high confidence и score около `0.82`;
- варианты: подтвердить, запросить одно изменение или передать на ручную
  обработку.

Второе сообщение:

> Подтверждаю план. Employee ID: EMP-DEMO-001. Выполни только утверждённые
> mock-шаги и покажи итог.

Во время исполнения ожидается компактный прогресс:

```text
Шаг 1 из 3 — выполнен
Шаг 2 из 3 — выполнен
Шаг 3 из 3 — выполнен
Исполнение завершено
```

## Последовательность Skills

1. `take_case` автоматически принимает `case_id` и управляет подготовкой.
2. Внутри него `load_case` загружает `VAL-DC-002`.
3. `find_case_scenarios` фильтрует каталог по теме/подтеме, применяет BM25 и
   возвращает до трёх сценариев с historical evidence.
4. `build_resolution_plan` один раз обращается к
   `z-ai/glm-5.2` через OpenRouter и валидирует план.
5. `review_resolution_plan` сохраняет решение `EMP-DEMO-001`.
6. `execute_approved_plan` последовательно вызывает:
   `check_account_state`, `request_expertise` с
   `account_balance_analysis`, затем
   `check_account_closure_eligibility`.

## Ожидаемый финал

- `execution_status=completed`;
- `zero_balance_confirmed`;
- `resolved_blockers=["negative_balance"]`;
- `eligible=true`;
- `remaining_blockers=[]`;
- `recommended_next_action=approve_case_closure`;
- реальное закрытие не выполняется;
- требуется финальное подтверждение сотрудника.

## Что проговорить голосом

- CasePilot работает через Skills Ouroboros; демонстрация проходит в
  локальном операторском интерфейсе, а Chat остаётся альтернативным способом
  проверки.
- Validation-кейс не содержит готового решения.
- Агент использует найденный сценарий, подтверждённый историческими кейсами,
  и ограничен каталогом экспертиз и строгой схемой плана.
- LLM только строит план; исполнение и точки остановки детерминированы.
- Даже после успешных проверок система не закрывает счёт самостоятельно.

## Последовательность кадров

1. Коротко показать `data/`: validation, history и каталог экспертиз.
2. Открыть Chat и отправить первое сообщение.
3. Показать понятное резюме кейса, найденный сценарий и план.
4. Отправить подтверждение сотрудника.
5. Показать прогресс трёх шагов.
6. Зафиксировать `eligible=true`, отсутствие блокеров и финальное решение за
   сотрудником.
7. Коротко показать runtime/audit trail как доказательство контролируемости.
