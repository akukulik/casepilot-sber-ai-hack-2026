# CasePilot: AS IS → TO BE и методика измерения

Актуально на 2026-07-20. Документ отделяет уже измеренные показатели от
гипотез, которые нужно подтвердить с операторами. Все кейсы синтетические.

## AS IS

Оператор вручную читает описание и историю общения, проверяет продукты и
системные признаки, выбирает последовательность проверок, запрашивает
экспертизы и формирует ответ клиенту. Репрезентативное банковское время
обработки организаторами или владельцем процесса проекту пока не передано,
поэтому CasePilot не заявляет выдуманную экономию в минутах или рублях.

## TO BE в MVP

CasePilot по `case_id`:

1. загружает кейс без готового решения;
2. ищет утверждённые сценарии;
3. строит план только из разрешённых действий и экспертиз;
4. ждёт подтверждения сотрудника;
5. выполняет синтетические проверки;
6. формирует evidence-bound рекомендацию и ответ клиенту.

Сотрудник сохраняет право остановить автоматизацию и решить кейс вручную.

## Уже измерено

На четырёх end-to-end кейсах план с `z-ai/glm-5.2`,
`reasoning_effort=medium`, был построен с первого запроса:

| Метрика | Результат |
|---|---:|
| Validation-кейсов в E2E-выборке | 4 |
| Планов, прошедших схему без repair | 4 из 4 |
| Терминальных веток | completed, manual_review, waiting_for_information |
| Provider/LLM calls | 57 |
| Суммарная стоимость | $0.4533 |
| Средняя стоимость полного агентного прогона | $0.1133 |
| Диапазон стоимости прогона | $0.0992–$0.1240 |

Источник: `docs/casepilot_demo_validation_report.md`. Это стоимость всех
учтённых Ouroboros-вызовов в прогоне, а не только планировщика.

## Что измерить перед презентацией

На 10–20 validation-кейсах:

- долю правильного top-1 сценария;
- recall@3 сценариев;
- долю планов без schema repair;
- долю планов, принятых сотрудником;
- долю остановок `manual_review` и `waiting_for_information`;
- медиану времени до готового плана;
- медиану времени полного mock-прогона;
- стоимость одного плана и полного прогона.

Для честного AS IS → TO BE нужны 3–5 операторов или владельцев процесса:

- замер ручного времени на одинаковой корзине кейсов;
- оценка полноты и применимости плана по шкале 1–5;
- число пропущенных обязательных проверок;
- число возвратов кейса на доработку.

## Формулы для презентации

После получения ручного baseline:

```text
time_saved_pct = (as_is_minutes - to_be_minutes) / as_is_minutes * 100
acceptance_rate = accepted_plans / generated_plans * 100
manual_review_rate = manual_review_runs / all_runs * 100
cost_per_accepted_plan = total_llm_cost / accepted_plans
```

До проведения такого замера показатели времени и ROI следует маркировать как
гипотезу, а не как доказанный эффект.
