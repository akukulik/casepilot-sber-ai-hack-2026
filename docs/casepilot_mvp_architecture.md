# Архитектура MVP CasePilot

## Назначение и границы

CasePilot хранит полностью синтетические отложенные кейсы направления
«Дебетовые карты»: карточные операции, возвраты, споры, банкоматы, регулярные
платежи, ограничения и закрытие счёта. В планировании и синтетическом
mock-исполнении участвуют 3 базовых и 17 расширенных MVP-сценариев. Реальные
банковские продукты не изменяются, клиенту ничего не отправляется.

## Данные

- `data/validation_cases.json` — входные кейсы без готового решения.
- `data/historical_cases.json` — 32 успешно решённых синтетических примера по
  широкому контуру обслуживания дебетовых карт.
- `data/expertise_catalog.json` — единственный разрешающий список из 20
  экспертиз для представленных подтем.
- `data/scenario_catalog.json` — утверждённые переиспользуемые сценарии с
  evidence-ссылками на исторические кейсы.
- JSON Schema в `schemas/` проверяют исходные данные, планы и runtime-записи.

Входной кейс включает идентификаторы, тему, описание, транскрибацию,
контекст клиента, продукты и синтетические системные признаки. Validation-файлы
не содержат `resolution_plan`, `expertise_results` или `final_result`.

## Поток

1. `load_case` находит validation-кейс по `case_id`.
2. `find_case_scenarios` фильтрует утверждённые сценарии по теме и подтеме,
   применяет BM25 и business rerank и возвращает до трёх сценариев.
3. `build_resolution_plan` передаёт кейс, сценарии с историческим evidence и каталог в
   `z-ai/glm-5.2` через OpenRouter с `reasoning_effort=medium`.
4. Ответ проходит строгую JSON Schema и программные проверки разрешённых
   действий, экспертиз и обязательных входов. Разрешён максимум один repair.
5. План версии 1 записывается в `data/runtime/plans.json` со статусом
   `proposed`.
6. `review_resolution_plan` принимает решение сотрудника:
   `approve_plan`, один `request_change` или `manual_review`.
7. Утверждённый план выполняет `execute_approved_plan`.
8. Для терминального исполнения `build_resolution_recommendation` формирует
   доказательное заключение, привязанное к фактическим шагам и `result_code`.

Отдельный управляемый контур анализирует expert-validated ручные решения,
создаёт scenario drafts, проводит deterministic offline replay и публикует
версию только после `EMP-DEMO-001`. Подробности:
[`scenario_evolution.md`](scenario_evolution.md).

## Skills и ответственность

- `load_case` — read-only загрузка кейса.
- `find_case_scenarios` — фильтр, BM25 и объяснимый business rerank без vector DB.
- `find_similar_cases` — legacy-инструмент, не используемый в основном потоке.
- `take_case` — единая операторская точка входа по `case_id`; объединяет
  новый независимый `run_id`, загрузку, retrieval, планирование и
  Chat-представление, но не исполняет план.
- `build_resolution_plan` — единственное LLM-планирование.
- `review_resolution_plan` — human-in-the-loop и версионирование.
- `check_account_state` — mock-состояние счёта.
- `check_pending_operations` — mock-проверка незавершённых операций.
- `request_expertise` — детерминированная mock-экспертиза по каталогу.
- `check_account_closure_eligibility` — итоговая детерминированная проверка.
- `case-actions` — пять универсальных синтетических действий расширенного
  контура.
- `execute_approved_plan` — строгая последовательная оркестрация разрешённых
  проверок, экспертиз и case actions; неизвестное действие останавливает
  исполнение.
- `build_resolution_recommendation` — одна генерация итоговой рекомендации
  через GLM 5.2, строгая Schema, evidence binding и безопасный fallback.
- `record_scenario_outcome` — сохраняет только экспертно подтверждённый
  успешный исход ручной работы или исправленного плана.
- `analyze_scenario_gaps` — draft из повторяющегося evidence-кластера.
- `validate_scenario_draft` — независимая проверка и offline replay.
- `review_scenario_draft` — экспертное approve/reject и runtime-публикация.

`openrouter_echo` — отдельный технический PoC провайдера и не входит в
бизнес-поток CasePilot.

## План, версии и human-in-the-loop

`plan_id`, `plan_version` и `status` находятся в runtime-обёртке, а бизнес-план
остаётся внутри поля `plan`. Для версии 1 разрешён один `request_change` и
создание версии 2 с `revision_context`. Для версии 2 повторное изменение
запрещено: кейс переводится в `manual_review`, версия 3 не создаётся.

Исполнение возможно только для точно совпадающей утверждённой версии.
Успешная eligibility-проверка также не закрывает продукт: результат
`approve_case_closure` требует отдельного финального решения сотрудника.

## Runtime и audit trail

- `runs.json` — независимые попытки обработки одного validation-кейса;
- `plans.json` — версии и статусы планов, связанные с `run_id`;
- `approvals.json` — решения и `employee_id`;
- `executions.json` — шаги, result codes, остановки и итог;
- `recommendations.json` — итоговые рекомендации, связанные с `execution_id`;
- `audit_log.jsonl` — append-only события создания, review, запуска, каждого
  шага и завершения/остановки.

Исходные validation-данные не изменяются. Всё изменяемое состояние находится
только в `data/runtime/`.

## Точки остановки

- отсутствует обязательный вход — `waiting_for_information`;
- шаг требует нового плана — `replan_required`;
- машинный result code требует человека — `manual_review`;
- неизвестное действие, неизвестная экспертиза или отклонение result code —
  исполнение останавливается;
- повторный `request_change` для версии 2 — `manual_review`.

После остановки последующие шаги не выполняются.

## Что имитируется

Состояние счёта, карточный процессинг, экспертизы, корректировка технической
комиссии, ограничения и eligibility полностью синтетические. OpenRouter и
GLM 5.2 используются для построения плана и операторской рекомендации; закрытие счёта,
изменение продуктов, отправка клиенту и интеграции с АБС отсутствуют.

## Ограничения MVP

Нет автономного самообучения, фоновых циклов, A2A, MCP, vector DB, реальных
банковских API и версии плана 3. Scenario evolution является
human-governed: draft не может опубликовать себя.
