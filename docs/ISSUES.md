# CasePilot: известные ограничения MVP

Актуально на 2026-07-20.

## До финальной демонстрации

### P1 — финальное решение по рекомендации не сохраняется для аналитики

Рекомендация строится и отображается, ответ клиенту редактируется, но
финальное действие сотрудника после рекомендации пока не записывается отдельным
runtime-событием. Нужен контракт вроде `approve_recommendation`,
`adjust_recommendation`, `manual_resolution`. Запись должна быть append-only,
связываться с `run_id` и `recommendation_id` и не изменять validation-данные.
Это позволит многократно демонстрировать один кейс без конфликтов.

## Осознанно исключено из MVP

- Редактирование предложенного плана и создание версии 2. В операторском MVP
  сотрудник либо подтверждает план, либо переводит кейс в ручную работу.
- Автоматическое продолжение после `waiting_for_information` и
  `replan_required`. CasePilot корректно останавливается и показывает
  необходимое действие; продолжение требует интеграции с банковским workflow.
- Версия плана 3, фоновые циклы и автоматическая публикация сценариев.

Backend-контракты ранних прототипов могут поддерживать больше состояний, но
они не заявляются как пользовательские функции текущего MVP.

## Ограничения демонстрационного контура

- Проверки и экспертизы детерминированные и синтетические.
- Реальные банковские API, БД, очередь кейсов и отправка сообщений отсутствуют.
- Frontend не имеет production-аутентификации. Полноценный агентный режим
  предназначен для localhost; GitHub Pages содержит только статический
  синтетический demo mode без API-ключа и банковских интеграций.
- Runtime использует локальные JSON-файлы и не рассчитан на несколько
  параллельных операторов.
- Scenario catalog сформирован на малой синтетической выборке; девять
  расширенных сценариев основаны только на одном historical-кейсе.
- Нет production-monitoring, трассировки SLA, миграций и disaster recovery.
- Scenario evolution запускается вручную; scheduler и экспертный UI отсутствуют.
- `find_similar_cases` является legacy-поиском и оставлен только для
  воспроизводимости ранней итерации.

## Не является дефектом

- Повторный запуск одного `case_id` создаёт новый независимый `run_id`.
- История runtime не поставляется в Git и начинается с чистого состояния.
- Реальный API-ключ хранится только в локальном Ouroboros.
- Презентационные задержки frontend не равны времени исполнения mock-Skills.
