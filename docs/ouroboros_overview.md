# Ouroboros: обзор перед разработкой CasePilot

Дата первоначальной проверки: 2026-07-16. Обновлено: 2026-07-19.

## Источники и границы достоверности

- Официальный гайд Sber AI Hack на приложенных фотографиях указывает Ouroboros 6.24.0 (июнь 2026), репозиторий `github.com/razzant/ouroboros` и установку macOS из `Ouroboros-6.24.0.dmg`.
- Официальный репозиторий был проверен напрямую. Стабильного тега `v6.24.0` в нём нет; доступны только `v6.24.0-rc.1` — `v6.24.0-rc.4`. Для изучения структуры использован ближайший реально существующий официальный тег `v6.24.0-rc.4`, commit `9bd7bb08821ad49ee13fea79afa7796ab1abeb3a`.
- 2026-07-19 организаторы отдельно разрешили использовать более новые официальные версии. После проверки релизов и уточнения из чата хакатона установлена стабильная версия 6.61.4. Подробная оценка находится в `docs/ouroboros_installation_assessment.md`.
- Фотографии гайда охватывают не все страницы и не определяют отдельный обязательный формат репозитория участника. Поэтому ниже разделены структура самого Ouroboros, его runtime-данные и предлагаемый минимальный каркас проекта. Неподтверждённые форматы не выдаются за требования.

## Основные понятия

### `SKILL.md`

Манифест и человекочитаемое описание навыка. В 6.24.0-rc.4 Skill — отдельный пакет одного из трёх типов:

- `instruction`: только инструкции Markdown;
- `script`: скрипты, запускаемые подпроцессом;
- `extension`: Python-плагин, регистрирующий инструменты, маршруты или виджеты.

`SKILL.md` может содержать YAML frontmatter. Официальная схема требует для полноценного манифеста `name`, `description`, `version`, `type`; в зависимости от типа добавляются `runtime`, `scripts` или `entry`, а также разрешения и таймаут. Парсер также допускает `skill.json` вместо frontmatter и body-only Markdown как instruction Skill. Значит, пример из фотографий с полями `type: script` и `entry: main.py` не полностью совпадает со схемой исследованного RC: для `script` официальная документация RC использует список `scripts`, а `entry` относится к `extension`. При разработке Skill нужно следовать схеме реально установленной сборки.

Исполняемые Skills проходят review, проверку хеша содержимого, явное включение владельцем, выдачу разрешений и проверку зависимостей. После отдельного разрешения пользователя создан минимальный PoC extension `openrouter_echo`; бизнес-Skill CasePilot ещё не создавался.

### `identity.md`

Постоянное самоописание агента: кто он, зачем существует и как понимает себя. Это изменяемый когнитивный артефакт, но сам файл защищён от физического удаления. Гайд хакатона требует заполнить его описанием сценария; делать это сейчас нельзя, поскольку установка и первичная настройка ещё не завершены.

### `knowledge/`

Структурированная долговременная база знаний в runtime-каталоге `~/Ouroboros/data/memory/knowledge/`. В исследованной версии туда входят, среди прочего, индекс/тематические файлы, реестр паттернов и improvement backlog. Для project-scoped задач факты проекта могут храниться отдельно в `data/projects/<project_id>/knowledge/`, чтобы не смешивать их с глобальной памятью.

### `scratchpad`

Рабочая память агента. Источником служит `scratchpad_blocks.json` (FIFO, максимум 10 блоков), а `scratchpad.md` автоматически генерируется для чтения. Вытесненные блоки журналируются. Это runtime-данные Ouroboros, а не файл исходного кода CasePilot.

### `dialogue_blocks`

`dialogue_blocks.json` — консолидированная блоками история диалога. Она формируется механизмом consolidation вместе с метаданными в `dialogue_meta.json`. Это runtime-память; вручную добавлять такой файл в проект CasePilot не требуется.

### MCP

Model Context Protocol — стандарт подключения внешних инструментов. В исследованной версии Ouroboros является MCP-клиентом для доверенных HTTP/SSE-серверов; MCP по умолчанию выключен, настраивается в Settings → Advanced, а найденные инструменты проходят обычную safety-проверку. Базовый runtime не является MCP-сервером. По заданию MCP для текущего каркаса и MVP пока не создаётся.

### A2A

Agent-to-Agent — взаимодействие нескольких специализированных агентов. В исследованной версии A2A вынесен в опциональный transport Skill из OuroborosHub, а не является обязательной частью базового проекта. Для первого MVP CasePilot мультиагентность не нужна и A2A не создаётся.

### Память агента

На первом запуске Ouroboros создаёт локальное дерево `~/Ouroboros/`. Ключевые области:

- `repo/` — локальный self-modifying Git-репозиторий Ouroboros;
- `data/state/` — состояние runtime, бюджет, очереди и состояние Skills;
- `data/memory/` — identity, scratchpad, профиль системы, knowledge и журналы;
- `data/logs/` — чат, события и вызовы инструментов;
- `data/uploads/` — вложения чата;
- `data/skills/` — пакеты Skills по источникам.

Память состоит не из одного файла: identity задаёт устойчивое самоописание, scratchpad хранит ближайший рабочий контекст, knowledge — долговременные структурированные знания, dialogue blocks — сжатую историю общения, журналы — происхождение изменений.

### Организация Skills и пользовательские Skills

Payload и состояние разделены:

- payload: `~/Ouroboros/data/skills/<bucket>/<name>/`;
- review/enable/grants/dependencies state: `~/Ouroboros/data/state/skills/<name>/`.

Buckets исследованной версии: `native`, `external`, `clawhub`, `ouroboroshub`. Пользовательские и вручную скопированные Skills должны лежать в `~/Ouroboros/data/skills/external/<name>/`. `native` зарезервирован для поставляемых приложением Skills.

## Какие файлы обязательны

### Для установленного Ouroboros

Официальный runtime сам создаёт `~/Ouroboros/repo/` и `~/Ouroboros/data/`. В памяти он гарантированно создаёт как минимум `identity.md`, `scratchpad.md`, `WORLD.md`, а также журналы; остальные runtime-файлы появляются по мере работы. Пользователю не следует вручную копировать эти файлы в репозиторий CasePilot.

### Для одного Skill

Skill — каталог с `SKILL.md` или `skill.json`. Для instruction Skill достаточно Markdown-манифеста/инструкции. Script и extension требуют дополнительных файлов, определённых манифестом.

### Для проекта участника

В доступной официальной документации не найден обязательный универсальный layout внешнего хакатон-проекта. Из гайда подтверждены будущие артефакты: описание AS IS, проектирование TO BE, `SKILL.md`, при необходимости MCP, политика автономии, метрики и README/код в публичном репозитории на этапе Project Results. Поэтому нельзя утверждать, что `identity.md`, `knowledge/`, `scratchpad` или `dialogue_blocks.json` должны лежать в корне CasePilot.

## 1. Как должен выглядеть проект хакатона

CasePilot остаётся отдельным репозиторием и не смешивается с runtime-кодом и
данными Ouroboros. Актуальная структура:

```text
uroboros_hackaton/
├── README.md
├── casepilot/      # runtime, review и mock-исполнение
├── data/           # синтетические кейсы, сценарии и runtime
├── frontend/       # операторский demo UI
├── schemas/        # JSON Schema
├── skills/         # канонические исходники внешних Skills
├── scripts/        # установка, очистка и тесты
├── tests/          # контрактный и интеграционный контур
└── docs/           # архитектура, workflow и отчёты
```

Это структура нашего приложения, а не обязательный стандарт Ouroboros.
`scripts/install_skills.py` копирует пакеты в пользовательский external-каталог
Ouroboros и создаёт локальные locator-файлы только в установленной копии.

## 2. Какие файлы нужны

- `README.md`, `LICENSE`, `SECURITY.md`, `CONTRIBUTING.md`;
- синтетические данные и JSON Schema;
- Skills с `SKILL.md`, кодом и минимальными grants;
- архитектура, operator workflow, demo script, Issues и TO BE;
- воспроизводимые команды установки, очистки и тестирования.

## 3. Возможности Ouroboros в MVP

- Chat, Skills и Settings для установки и диагностики.
- Набор узких Skills с единой точкой входа `take_case`.
- Память и контекст: `identity.md`, scratchpad и knowledge — как runtime-возможности, без копирования служебных файлов в проект.
- Safety layer и Advisory review, рекомендованный гайдом для хакатона.
- Human-in-the-loop для критичных решений.
- OpenRouter и `z-ai/glm-5.2` для плана и итоговой рекомендации.

## 4. Что не используется в MVP

- A2A и несколько агентов.
- MCP и реальные банковские интеграции.
- Самоэволюция `/evolve`.
- Background consciousness `/bg`.
- Локальная модель.
- Widgets, marketplace publishing, transports и автономные расписания.
- Vector DB: для текущего объёма достаточно фильтра, BM25 и rerank.
