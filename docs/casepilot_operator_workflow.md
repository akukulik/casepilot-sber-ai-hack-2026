# Операторский workflow CasePilot

## Единая точка входа

Оператор или будущий frontend передаёт только `case_id` в Skill `take_case`.
В Chat достаточно написать `VAL-DC-002` или «Возьми кейс VAL-DC-002».
Каждый такой ввод создаёт новый независимый `run_id`, поэтому validation-кейс
можно тестировать повторно. Явная просьба открыть последний запуск использует
`run_mode="latest"` и не вызывает LLM.

Зарегистрированное описание инструмента автоматически сообщает агенту его
роль: загрузить кейс, найти утверждённые сценарии, построить план и показать
операторскую карточку. Глобальный `memory/identity.md` Ouroboros не изменяется,
поэтому CasePilot не влияет на другие сценарии приложения.

`take_case` выполняет:

1. `load_case`;
2. точный фильтр scenario catalog по `case_topic` и `case_subtopic`;
3. BM25 и business rerank;
4. выбор до трёх сценариев с historical evidence;
5. `build_resolution_plan` через GLM 5.2;
6. сохранение proposed-плана;
7. формирование `operator_view` и читаемого `operator_message`.

Skill не подтверждает и не исполняет план.

## Retrieval

Основной фильтр требует совпадения темы и подтемы. Если кандидатов нет,
используется контролируемый fallback: сначала совпадение темы, затем вся
история.

Внутри выборки score состоит из:

- совпадение причины блокировки — 0.45;
- нормализованный BM25 — 0.35;
- пересечение типов продуктов — 0.15;
- совпадение приоритета — 0.05.

Планировщик получает максимум три сценария и связанные исторические evidence-кейсы.

Текущая JSON-база подходит для MVP. При существенном росте истории тот же
контракт можно сохранить, заменив полный проход по файлу на предварительно
построенный BM25-индекс. Vector search не требуется до появления измеримой
нехватки качества.

## Операторский ответ

Chat показывает:

- ID, тему, дату, приоритет и описание;
- понимание проблемы и признаки;
- основной сценарий;
- стратегию из 2–20 шагов;
- необходимые экспертизы;
- confidence;
- три действия сотрудника.

Служебный JSON остаётся доступен будущему frontend, но не показывается в
основном сообщении.

## Короткие команды

- `Подтверждаю` → `approve_plan`;
- `Изменить: <комментарий>` → `request_change`;
- `Беру вручную` → `manual_review`.

Chat берёт `case_id`, `plan_id` и `plan_version` из последнего результата
`take_case`. Для MVP используется `EMP-DEMO-001`.

Команды показываются только для proposed-плана. Последний утверждённый,
исполненный или остановленный запуск можно явно открыть read-only.
